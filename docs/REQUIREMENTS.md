## 1. 앱 목적

Memory Trip은 여행과 캠핑의 경험을 **사진·장소·감정·비용**과 함께 기록하고
시간이 지나도 당시의 분위기를 다시 떠올릴 수 있도록 돕는 개인 아카이브 앱이다.

핵심 목적
- 여행/캠핑 경험의 사진 중심 아카이브
- 장소 기반 회고
- 감정 기록
- 비용 기록
- 미래 여행을 위한 회고 자료 제공

## 2. 여행 형태

여행 기록은 형태에 따라 입력받는 항목이 달라진다.

### 2.1 Phase 1 — 지원 형태

| 형태 | 설명 | 전용 입력 |
|---|---|---|
| `trip` (여행) | 일반 여행 (국내/해외) | 숙소, 식당, 이동 정보 |
| `camping` (캠핑) | 캠핑 중심 활동 | 캠핑장 정보, 장비 기록 |

### 2.2 Phase 2+ — 확장 예정 형태

아래 형태는 Phase 1 단계에서는 구현하지 않으나, **DB/UI/타입 구조는 확장 가능하도록 설계**한다.

| 형태 | 핵심 데이터 | 도입 시점(잠정) |
|---|---|---|
| `hiking` (등산/하이킹) | 코스명, 거리, 누적 고도, 난이도, 정상 도달 여부 | Phase 2 |
| `car_camping` (차박) | 차량 정보, 박지 위치, 평탄도, 안전성 | Phase 2 |
| `cycling` (자전거 여행) | 일자별 라이딩 거리, 누적 고도, 평균 속도 | Phase 3 |
| `workation` (워케이션/한달살기) | 장기 체류, 작업 환경(WiFi/책상/카페), 월세/생활비 | Phase 3 |
| `business` (출장) | 미팅 일정, 동행 동료, 영수증/정산 | Phase 3+ |

### 2.3 확장 가능성을 위한 설계 원칙

Phase 2 이후의 형태 추가가 **기존 데이터 마이그레이션 없이** 가능하도록 다음 원칙을 따른다.

1. **`trip_type` 컬럼은 string 형으로 저장** (SQLite에 enum 제약 X). 코드에서만 union literal type 으로 좁힌다.
   ```ts
   type TripType = 'trip' | 'camping'; // Phase 1
   // Phase 2: 'trip' | 'camping' | 'hiking' | 'car_camping'
   ```
2. **형태별 전용 데이터는 별도 테이블로 분리** (e.g. `trip_lodging`, `camping_site`).
   추후 `hiking_course`, `cycling_log` 같은 테이블만 추가하면 되고 기존 테이블 변경 없음.
3. **공통 필드(3.1)는 `trips` 한 테이블에서 관리.** 형태가 늘어도 공통 부분은 그대로.
4. **여행 생성/수정 화면은 폼 분기 구조로 작성.** 형태 enum 값에 따라 동적으로 추가 폼을 끼워 넣는 방식 (스위치 분기 X, 등록형 컴포넌트 맵 사용 권장).
5. **`stores/`, `services/`는 형태에 무관한 추상화 우선.** 형태별 로직이 필요하면 strategy 패턴으로 분리.

> 동일 여행 안에서 두 모드를 혼합할 수 있는지 (예: 여행 중 1박만 캠핑) 는 Phase 2에서 결정. Phase 1은 단일 형태로 가정.

## 4. 화면 구성

### 4.1 홈 화면

#### 4.1.1 메인 CTA
- **"출발" 큰 버튼** (✈ 아이콘 + "새 여행 시작하기" 보조 라벨)
- 클릭 시 바텀 시트 모달 → 3필드 입력 → 진행 중 상세 화면 직행 (§4.4 참조)

#### 4.1.2 보조 진입
- "사진으로 여행 만들기" — 갤러리에서 사진 선택 (Phase 1)

#### 4.1.3 상태별 동적 영역 (메인 CTA 위 또는 아래)

**진행 중 여행 카드** (있을 때만) — hero placeholder/사진 + 빠른 추가 5버튼
- 정의: end_date NULL 또는 today 이전이고, **최근 14일 이내 활동**이 있는 여행
- 14일 동안 활동 없는 여행은 여기서 안 보임 → 목록에서 "종료일 미정" 배지로만 노출

**예정 여행 카드** (있을 때만) — start_date 미래
- 카드 액션: "출발" → 진행 중으로 전환 (start_date를 today로 변경하거나 그대로 두고 활동 시작)

> "이 여행 끝났나요?" 같은 마무리 알림 카드는 두지 않는다. 종료일 미정 여행은 §4.2 목록에서 시각적으로 강조해 사용자가 인지하도록 한다.

#### 4.1.4 회고 영역 (스크롤)
- **최근 여행 (3~5건)** — 카드형 (대표 사진 + 제목 + 기간 + 상태 배지 + 평점/사진 수)
  - 상태 배지는 §4.2의 4가지 규칙과 동일 (예정 / 진행 중 / 종료일 미정 / 완료)
  - **종료일 미정 카드는 점선 테두리 + amber 배지로 시각적 강조** — 목록과 동일 처리
- 통계 — "12개 여행 · 87곳 장소 · 1,234장 사진 · 38박"
- (Phase 2+) "작년 오늘" 카드

### 4.2 여행 목록 화면

**모든 여행 상태(예정/진행 중/종료일 미정/완료)의 단일 진입점**. 종료일 미정 여행도 여기서 시각적으로 인지하고 정리.

#### 4.2.1 헤더 + 필터

- 헤더: "여행 목록" + ⌕ 검색 + ⚙ 필터 상세
- 필터 칩 (가로 스크롤): `[전체] [trip] [camping] [상태 ▾] [연도 ▾]`
- 표시: "N건 · 최근순 ▾"

#### 4.2.2 카드 구성

| 영역 | 내용 |
|---|---|
| 좌측 (80×80) | 대표 사진 썸네일 (없으면 형태별 placeholder) |
| 우측 상단 | 제목 + **상태 배지** (우측 끝) |
| 우측 중간 | 기간 + 형태 |
| 우측 하단 | 평점 · 사진 수 · 총 비용 (또는 형태별 메타) |

#### 4.2.3 상태 배지 (4가지)

| 상태 derive | 배지 텍스트 | 색상 | 카드 강조 |
|---|---|---|---|
| `start_date > today` | `예정 D-N` | c-blue | 일반 카드 |
| `start_date <= today`, `end_date >= today 또는 NULL`, **최근 14일 활동** | `● 진행 중` | c-amber | 일반 카드 |
| `end_date IS NULL`, **최근 14일 활동 없음** | `📅 종료일?` | c-amber (파일) | **점선 테두리 + amber 배지** |
| `end_date < today` | (배지 없음) | — | 일반 카드 |

> "종료일 미정"은 사용자에게 "정리할 거 있네" 인지시키는 것이 목적. 클릭하면 상세로 진입해서 종료일 입력 또는 "여행 도장 짓기" 화면으로.

#### 4.2.4 정렬 / 필터

- **기본 정렬**: 최근 수정/생성순 (default)
- 다른 정렬 옵션: 시작일순 / 평점순
- 필터 조합: 형태 + 상태 + 연도 + (Phase 2+) 동행자/태그

#### 4.2.5 검색 (Phase 1 — 키워드)

헤더 ⌕ 탭 → 검색 모드. 다음 필드에서 키워드 매칭:
- Trip.title, summary
- Place.name
- DiaryEntry.memo, best_moment, want_to_revisit
- Tag.name
- Trip.companions (JSON 검색)

(Phase 3+ 자연어 검색은 별도 인덱스/임베딩 검토)

#### 4.2.6 빈 상태

- 여행 0건 → "첫 여행을 시작해보세요" 메시지 + 출발 CTA

### 4.3 여행 상세 화면

상태에 따라 두 가지 모드.

#### 4.3.1 진행 중 모드 (status = in_progress)
- 헤더: 뒤로 + 여행 제목 + 더보기 메뉴
- 상태 pill: "● 진행 중 · N일째"
- Hero 영역
  - 사진 0장 → 그라데이션 placeholder + "첫 기록을 남겨보세요"
  - 사진 1장+ → 가장 최근 사진 (탭하면 갤러리)
- **빠른 추가 5버튼** — 📸 카메라 / 🖼 갤러리 / 📍 장소 / 💰 비용 / 📝 일기
- 타임라인 (시간순 이벤트 리스트, 비어있으면 안내)

#### 4.3.2 완료 모드 (status = completed)
- 대표 사진 (히어로 영역)
- 타임라인 (일자별 이벤트)
- 사진 갤러리
- 비용 요약
- 감정 기록 (DiaryEntry)
- 지도 (해당 여행 루트)
- 만족도, 한 줄 요약, best_moment 등 회고 데이터

#### 4.3.3 빠른 추가 화면 (카메라 / 갤러리 / 장소 / 비용 / 일기)

진행 중 카드의 5버튼(또는 진행 중 상세의 동일 버튼)에서 진입하는 풀스크린 폼들.

**공통 패턴**
- 헤더 일관: `[취소] | 화면명 | [저장]`
- 저장 후 → 진행 중 상세 화면으로 복귀
- 모든 저장은 **단일 SQLite 트랜잭션** (롤백 안전)
- 사진/장소 저장 시 backup_queue 자동 등록

##### 4.3.3.1 카메라 추가 (📸)

**진입**: 📸 버튼 → 시스템 카메라 → 촬영 → **미리보기 풀스크린** (§4.3.3.3 공통)

**특성**
- 결과 사진 1장 (보통). 연속 촬영 모드면 N장.
- 위치 출처: **현재 GPS** (촬영 직후 즉시 측정)
- `Photo.location_source = 'gps'` (또는 EXIF에 좌표 들어가면 `'exif'`)
- 즉석 메모 자연스러움 (방금 찍은 순간의 감정 그대로)
- (Phase 2+) 현재 위치 기반 **근처 Place 자동 제안** ("성산일출봉 게스트하우스에서 찍으셨나요?")

##### 4.3.3.2 갤러리 추가 (🖼)

**진입**: 🖼 버튼 → 시스템 사진 picker (다중 선택) → **미리보기 풀스크린** (§4.3.3.3 공통)

**특성**
- 결과 사진 다중 가능 (한 번에 N장)
- 위치 출처: **EXIF** (촬영 당시 좌표)
- `Photo.location_source = 'exif'` (EXIF에 좌표 없으면 NULL)
- 사후 메모 (회고하면서 작성)
- (Phase 2+) EXIF 좌표 기반 **가까운 Place 자동 제안**

##### 4.3.3.3 사진 미리보기 (카메라 / 갤러리 공통)

카메라·갤러리 양쪽 모두 동일한 미리보기 풀스크린으로 수렴. 1장이든 N장이든 UI 동일.

**구성**
- 썸네일 그리드 (3열). 각 썸네일에 촬영 시간 작게.
- **일괄 옵션** (모든 사진에 동일 적용):
  - 메모
  - 장소 연결 (이번 여행의 Place 드롭다운, 또는 "연결 안 함")
- 자동 감지된 정보 표시: 촬영 시간 범위 / 위치 출처(EXIF/GPS) / 백업 대상
- 개별 편집은 저장 후 갤러리에서

**저장 동작 (트랜잭션)**
- 각 사진을 `${doc}photos/{tripId}/{photoId}.jpg` 로 복사
- Photo 행 N개 일괄 INSERT
  - `taken_at` / `gps_lat`/`gps_lng` / `location_source` ← 출처에 따라
  - `place_id` / `memo` ← 일괄값
- backup_queue 일괄 등록

##### 4.3.3.4 장소 추가

**진입**: 📍 버튼 → **타입 선택 바텀 시트** → 풀스크린 폼

**타입 선택 시트** (trip_type에 따라 옵션 다름)
| trip_type | 표시 옵션 |
|---|---|
| `trip` | [숙소] [식당] |
| `camping` | [캠핑장] [식당] |

> Phase 2+ "추가 정보 토글"로 다른 타입도 노출 가능.

**폼 구조 (4섹션, 접고 펼치기)**

| 섹션 | 필드 | 기본 상태 |
|---|---|---|
| 기본 정보 | 이름(필수) / 방문일(default today) / 위치(자동/수동) | 펼침 |
| 비용 | 가격 (NULL 허용) → 입력 시 자동 Expense 생성 | 펼침 |
| 평가 | rating + 타입별 세부 점수 + pros/cons + 재방문 | 접힘 |
| 메모 | 자유 텍스트 | 접힘 |

**저장 동작 (트랜잭션)**
- Place 행 INSERT
- xxx_details (LodgingDetails / RestaurantDetails / CampsiteDetails) INSERT
- 가격 입력됐으면 Expense INSERT (place_id 연결)

##### 4.3.3.5 비용 추가

**진입**: 💰 버튼 → 풀스크린

**구성**
- **카테고리 선택** (큰 아이콘 2x4 그리드): 숙박 / 교통 / 식비 / 카페 / 쇼핑 / 캠핑장 / 장비 / 기타
- **금액 입력** — 큰 디스플레이 + 숫자 키패드 (가계부 앱 패턴, 한 손 빠른 입력)
- 보조 (접힘, 펼치면 노출):
  - 일자 (default today)
  - 장소 연결 (이번 여행의 Place 드롭다운)
  - 메모

**자동 카테고리 추천**: Phase 2 이후 결정 (현재는 사용자가 매번 수동 선택)

**저장 동작**: Expense 행 1개 INSERT

##### 4.3.3.6 일기 추가

**진입**: 📝 버튼 → 풀스크린

**구성**
- 일자 선택 (default today)
- 토글 **"전체 회고로 기록"** → date NULL 모드 전환
  - 이 모드에서만 `best_moment`, `want_to_revisit` 입력란 노출
- **감정 칩** (다중 선택, 7개): 행복 / 힐링 / 감동 / 설렘 / 피곤 / 외로움 / 아쉬움
- 메모 (큰 텍스트 영역)

**저장 동작**
- `(trip_id, entry_date)` UNIQUE 제약
- 같은 날 일기 이미 있으면 → **"업데이트하시겠어요?" 확인 다이얼로그** → UPDATE
- 없으면 → INSERT
- 사용자는 하루에 일기 1건만 (덮어쓰기 패턴)

### 4.4 여행 생성 / 수정 화면

#### 4.4.1 진입점

기본은 **단일 진입점**. 홈 화면 메인 CTA **"출발"** 한 곳에서 모든 케이스를 흡수한다.

| 사용자 시점 | 시작일 입력값 | 결과 |
|---|---|---|
| 출발 전 등록 | 미래 날짜 | 예정 여행 |
| 출발일/도착일 등록 | today (default) | 진행 중 |
| 사후 등록 | 과거 날짜 | 진행 중 또는 완료 |

→ 진입점은 하나, **시작일 입력값으로 모든 케이스 자연 처리**.

| 보조 진입점 | 위치 | Phase |
|---|---|---|
| 사진으로 만들기 | 사진 갤러리에서 선택 후 액션. EXIF로 start_date 추정 | Phase 1 |
| 자동 제안 | 평소와 다른 위치 사진 → "여행 만드시겠어요?" | Phase 2+ |

#### 4.4.2 빠른 생성 모달 (바텀 시트)

"출발" 클릭 → **하단에서 올라오는 바텀 시트 모달**. 배경 화면(홈)이 살짝 보이는 상태.

**입력 필드 — 3개만**
1. 여행 제목 (필수, 텍스트 입력)
2. 여행 형태 (필수, 토글: `trip` / `camping`)
3. 시작일 (필수, default = today, 탭하면 날짜 피커)

**액션**
- `[취소]` — 모달 닫기, 데이터 폐기
- `[✈ 시작하기]` — Trip 생성 트랜잭션 → **진행 중 상세 화면 직행** (홈 복귀 X)

**보조 안내**
- "종료일은 마무리할 때 입력해도 돼요" — 부담 줄이는 텍스트
- 2탭(제목 + 시작하기)으로 완료 가능

#### 4.4.3 생성 후 landing — 진행 중 상세 화면 직행

저장 즉시 진행 중 상세 화면(§4.3 참조)으로 이동.
- 큰 hero placeholder ("첫 기록을 남겨보세요")
- 빠른 추가 5버튼 (📸 카메라 / 🖼 갤러리 / 📍 장소 / 💰 비용 / 📝 일기) — 한 탭으로 즉시 추가 가능
- 타임라인 (비어있음)

홈으로 돌아가도 진행 중 카드가 표시되지만, **만든 직후엔 상세 화면이 default landing**.

#### 4.4.4 전체 폼 (수정 또는 상세 입력 시)

진행 중 상세 화면 → "여행 정보 편집" 메뉴 → 전체 폼 진입.
- 기본 정보 입력 (3.1) — 모든 필드 노출
- 여행 형태 선택 → 형태에 맞는 추가 입력 폼 분기
- 사진 업로드
- 숙소 / 식당 / 캠핑 정보 입력 (형태에 따라)

### 4.4-A 여행 도장 짓기 화면

여행 종료 시점에 자동 통계와 함께 회고를 입력하는 **선택적** 화면.

#### 4.4-A.1 정체성

- **status는 종료일만으로 결정** (DOMAIN_MODEL §0 #11). 종료일 입력 = "끝남" 선언.
- 본 화면의 satisfaction/summary/회고는 **모두 선택**. 입력 안 해도 완료 상태가 됨.
- 화면의 의의는 의무 마무리가 아니라 **회고 경험 제공** + 자동 통계로 추억 자극.

#### 4.4-A.2 진입 트리거

| 시점 | 동작 |
|---|---|
| 종료일 입력 직후 (편집 화면 또는 본 화면에서) | "회고를 남겨보세요" 알림 → 본 화면 진입 (옵션) |
| 목록(§4.2)에서 종료일 미정 카드 클릭 | 상세로 진입 → 메뉴에서 "도장 짓기" 또는 종료일만 입력 |
| 사용자 자발적 | 진행 중/완료 상세 → 메뉴 "도장 짓기" |

> 홈에 마무리 알림 카드는 두지 않는다. 목록의 시각적 강조(점선+amber 배지)로 충분.

#### 4.4-A.3 화면 패턴

**스크롤 풀스크린 — 요약 카드 + 인라인 입력**

자동 통계 카드가 위에서 추억을 자극 → 자연스럽게 입력란 채우게 됨. 모두 선택, 언제든 건너뛰기.

#### 4.4-A.4 구성 (8섹션)

1. 헤더: `[건너뛰기] | 도장 짓기 | [완료]` — [완료]가 곧 저장
2. **여행 요약 카드** (자동 통계 — c-teal 강조 카드)
   - 제목 + 기간 + 형태
   - 박/일수 / 방문 장소 N / 사진 N / 총 비용
   - 1일 평균 / 가장 큰 지출 카테고리
3. **종료일 확정** — NULL이었으면 today 제안. 이미 있으면 변경 가능
4. **이번 여행 어땠어요?** — 큰 별 5개 (rating)
5. **한 줄로 표현하면** — placeholder "비 와도 좋았던 제주"
6. **가장 기억에 남는 순간** — 텍스트 영역 (DiaryEntry.best_moment)
7. **다시 가고 싶은 곳** — 텍스트 영역 (DiaryEntry.want_to_revisit)
8. **대표 사진 선택** — 사진 그리드. 안 고르면 가장 최근 사진 default. (Phase 2+ 자동 추천)

> "✓ 마무리하기" 같은 별도 큰 버튼은 **없음**. 헤더 [완료]가 저장.

#### 4.4-A.5 저장 동작 (단일 트랜잭션)

- `Trip.end_date`, `satisfaction`, `summary`, `hero_photo_id` UPDATE
- `DiaryEntry` (trip_id, entry_date NULL, best_moment, want_to_revisit) INSERT/UPDATE
  - 기존 trip-level DiaryEntry 있으면 UPDATE, 없으면 INSERT (UNIQUE 제약)
- 저장 후 → **완료 모드 상세 화면 (§4.3.2) 자동 이동**

#### 4.4-A.6 재진입

회고 데이터는 언제든 다시 진입해서 수정 가능. "건너뛰기"로 닫아도 데이터 손실 없음 (입력한 만큼 저장).

#### 4.9.2 5개 섹션 구성

##### 4.9.2.1 백업 및 복구

본 앱이 로컬 SQLite 기반이라 가장 핵심 섹션.

| 항목 | 동작 |
|---|---|
| 자동 백업 | 토글 (default 켜짐) |
| 백업 위치 | 페이지 → **iCloud Drive** (iOS) / **Google Drive** (Android) — 변경 가능. Phase 2+ Dropbox/OneDrive 등 추가 검토 |
| 마지막 백업 | 표시만 — `memory_trip_backup_YYYYMMDD.db · 9.4 MB` |
| 지금 백업하기 | 버튼 → 즉시 SQLite 백업 + 클라우드 업로드 |
| 백업 목록 | 페이지 → 보관된 백업들 (날짜별, 크기) |
| 백업에서 복구 | 페이지 → 복구할 백업 선택 → 확인 다이얼로그 → 실행 |
| **DB 무결성 검사** | 표시 (`정상 ✓` / `이상 ✗`). 탭하면 재검사 실행. 자동 검사는 백그라운드 주기적으로 수행 |

##### 4.9.2.2 저장소

| 항목 | 표시 |
|---|---|
| 사용 중 | 총 사용량 + **스택 바 시각화** (사진/DB/백업 구분 색) |
| ㄴ 사진 | 248 MB |
| ㄴ 데이터베이스 | 12 MB |
| ㄴ 백업 | 82 MB |
| 정리 도구 | (Phase 2+) — 비활성 표시 |

##### 4.9.2.3 권한

각 권한 상태 표시 + 탭하면 시스템 설정으로 이동 (앱에서 직접 수정 불가).

| 권한 | 가능한 상태 |
|---|---|
| 위치 | 허용됨 / 거부됨 / 요청 안 됨 |
| 카메라 | 동일 |
| 사진 | 동일 |
| 알림 | 동일 |

거부됨 상태는 빨간색으로 강조, 허용됨은 초록색.

##### 4.9.2.4 앱

| 항목 | 옵션 |

## 5. 우선순위

|---|---|
| **Phase 1 (MVP)** | 여행 형태 `trip` + `camping` 완전 지원 / 3.1 (공통) / 3.2 (사진) / 3.3 (여행 모드) / 3.4 (캠핑 모드) / 3.5 (비용) / 3.6 (감정 — DiaryEntry) / 4.1 (홈) / 4.2 (목록) / 4.3 (상세) / 4.3.3 (빠른 추가 5개) / 4.4 (생성·편집) / 4.4-A (도장 짓기) / 4.5 (갤러리) / 4.9 (설정) |

### 5.1 Phase 1 완료 기준 (Definition of Done)

- `trip` / `camping` 두 형태 모두 생성·수정·삭제 가능
- 사진 업로드 + SQLite 메타 + 로컬 백업 큐 등록까지 동작
- 비용 카테고리 입력 + 총비용/1일 평균 자동 계산
- 홈/목록/상세/생성 4개 화면 동작
- 오프라인 상태에서 위 모든 기능 동작
- 자동 로컬 SQLite 백업 (보관 정책은 [CLAUDE.md §6](../CLAUDE.md) 참조)

## 6. 미정/추후 결정 항목

- 단일 여행 안에서 여러 형태(`trip` ↔ `camping`, `camping` ↔ `hiking` 등) 혼합 가능 여부 → Phase 4
- 동행자 필드를 자유 텍스트로 둘지, 별도 동행자 엔티티로 정규화할지
- 사진 위치는 EXIF 우선인지 사용자 입력 우선인지
- 비용 통화 단위 (단일 통화 / 다중 통화 환전)
- 검색의 자연어 처리 수준 (단순 키워드 매칭 vs LLM/임베딩 기반)
- 공유 기능 여부 (현재는 개인 아카이브 전제)
- Phase 2+ 신규 여행 형태(`hiking`/`car_camping` 등) 의 상세 필드 정의 (도입 직전 별도 라운드로 확정)
